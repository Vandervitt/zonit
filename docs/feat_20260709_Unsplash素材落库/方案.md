# 方案：Unsplash 素材下载落库（素材库页 + 编辑器）

- 分支：`feat_20260709_Unsplash素材落库`
- 日期：2026-07-09
- 状态：已通过设计审批，待写实现计划

## 一句话

给「个人工作室没有现成图」的用户提供 Unsplash 作为媒体来源：在**素材库页**和**编辑器选图弹窗**里搜 Unsplash，选中的图**下载存进用户自己的 Vercel Blob**、成为一条真实的自有 `MediaItem`（不是热链），署名元数据入库并仅在后台素材库展示。

## 已锁定决策

| 决策点 | 选择 | 理由 |
|---|---|---|
| 图片形态 | **下载存 Blob**（非热链） | 消除裂图风险、加载走自有 CDN、素材库语义干净、复用现有上传管线代码更省 |
| 署名展示 | 元数据入库 + **仅后台素材库卡片**展示，公开落地页不强挂 | 兼顾 Unsplash 署名义务与客户营销页 UX |
| 改动范围 | **素材库页 + 编辑器**两处统一 | 两处共用同一 import 端点，行为一致、所有 Unsplash 图都变自有 |
| 下载尺寸 | `regular`（~1080px 宽） | 落地页首图/分区图足够，体积可控 |
| 搜索 | 复用现有 `GET /api/unsplash/search`（扩展返回字段） | 不重复造轮子 |

## Unsplash API 合规（两条强制项，落库同样适用）

1. **触发下载**：用户「选用」一张图时，服务端必须 `GET` 该图的 `links.download_location`（带 `Client-ID`）。纯后台、无 UI 影响。
2. **署名**：存摄影师名 + 主页链接（含官方要求的 `?utm_source=<app>&utm_medium=referral` 参数），在素材库卡片展示。
3. 生产需申请 production key（demo key 限 50 次/小时）；未配 key 时搜索走空态。

## 架构（按单元，各自单一职责）

### 1. 数据层

**迁移 `migrations/017_add_media_attribution.js`**（幂等）：`media` 表加三列
- `source TEXT NOT NULL DEFAULT 'upload'`（取值 `'upload'` | `'unsplash'`）
- `credit_name TEXT`（摄影师名，可空）
- `credit_url TEXT`（摄影师 Unsplash 主页链接，可空）

**`lib/media-db.ts`**
- `MediaItem` 接口加 `source: 'upload' | 'unsplash'`、`creditName: string | null`、`creditUrl: string | null`
- `rowToMediaItem` 映射新列
- `insertMedia` 增加可选参数 `attribution?: { source; creditName; creditUrl }`；不传时默认 `source='upload'`，credit 为 null——**现有上传路径零改动**

### 2. 服务层（两个消费方共用）

**扩展 `GET /api/unsplash/search`**：结果每项补
- `downloadLocation`（来自 `p.links.download_location`，供合规触发）
- `user.profileUrl`（`https://unsplash.com/@{username}?utm_source=<app>&utm_medium=referral`）

**新 `POST /api/media/unsplash`**
- 入参：`{ downloadLocation, imageUrl (regular), creditName, creditUrl, alt? }`
- 流程：
  1. 鉴权（复用现有 session 校验，需登录）
  2. `GET downloadLocation`（`Client-ID`）触发 Unsplash 下载计数（失败仅记录警告，不阻断）
  3. `fetch(imageUrl)` 取字节
  4. `put()` 存 Blob（复用 `lib/media-upload` 逻辑）
  5. `insertMedia(userId, blobUrl, filename, 'image', size, { source:'unsplash', creditName, creditUrl })`
  6. 返回新 `MediaItem`
- **顺序保证**：Blob 落地成功后才插库，任一步失败即中止，不留半条脏记录

### 3. UI 层（两个消费方，共用上面的端点）

**A · 素材库页 `app/admin/(workspace)/media/page.tsx`**
- header「上传」旁加**「从 Unsplash 添加」**按钮 → 打开 antd Modal
- Modal：搜索框 + 结果网格（每图下摄影师名）→ 点某图 → `POST /api/media/unsplash` → 成功后 `mutate` 把新 MediaItem 前插进网格 → 该图立即出现在素材库
- 下载中该图卡显示 loading，防重复点击

**B · 编辑器 `landing-editor/ui/media/UnsplashTab.tsx`**（改现有热链行为）
- 点某图不再 `onChange(unsplashUrl, alt)`，而是：`POST /api/media/unsplash` → 拿回 `MediaItem.url`（Blob）→ `onChange(blobUrl, alt)` → 关闭弹窗
- 图同时进用户素材库（编辑器 library tab / 素材库页都能复用）
- 加下载中 loading 态（原本瞬时，现在多一次下载往返）

**C · `components/media/MediaGrid`（素材卡片）**
- `source==='unsplash'` 的卡片显示「© {creditName} · Unsplash」角标，可点到 `creditUrl`

**无障碍**：新增交互元素（Unsplash 按钮、搜索框、结果图、角标链接）aria-label 走 i18n key，满足项目 E2E 定位链。

## 数据流

```
搜索框输入 → GET /api/unsplash/search（复用+扩展）→ 结果网格
   → 用户点选某图
   → POST /api/media/unsplash { downloadLocation, imageUrl(regular), credit… }
       服务端：ping download_location → fetch bytes → put() Blob → insertMedia(source=unsplash)
   → 返回 MediaItem
   → 素材库页：mutate 前插进网格；编辑器：onChange(blobUrl, alt) 填字段
```

## 错误处理

| 场景 | 处理 |
|---|---|
| 未配 `UNSPLASH_ACCESS_KEY` | 搜索返回空 + 提示「未配置 Unsplash」 |
| Unsplash 429 限流 | 提示「请求过于频繁，稍后再试」 |
| download_location 触发失败 | 记录警告，不阻断（不影响落库） |
| 图片 fetch / Blob put 失败 | toast 报错，不插库、不留脏记录 |

## 测试

- **单测**（`app/api/media/unsplash` handler）：mock fetch，断言
  - 触发了 `download_location`
  - `insertMedia` 收到 `source='unsplash'` 及正确 creditName/creditUrl
  - 上游失败时不调用 `insertMedia`
- **E2E**：
  - ① 素材库页开 Unsplash Modal → 搜 → 点图 → 网格出现新素材（demo/mock 数据）
  - ② 编辑器 Unsplash tab 点图 → 字段填入 Blob url（断言非 `images.unsplash.com` 域）
  - 定位走 i18n key → aria-label

## 明确不做（YAGNI）

- 编辑器/素材库以外的第三处入口
- 无限滚动分页（首屏 `per_page` 够用，「加载更多」留二期）
- 收藏 / 分类 / 标签
- Unsplash 视频
