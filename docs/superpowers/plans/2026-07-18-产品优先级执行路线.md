# 产品优先级执行路线（2026-07-18）

> **For agentic workers:** 本文档是**主路线图**，按优先级排列 4 个执行阶段。每个开发阶段开工前，须按 `superpowers:writing-plans` 另写该阶段的详细实现计划（bite-sized TDD 任务），经用户确认后再实现。本文档锁定的是范围、契约方向与验收标准。

**目标：** 把已上线但未经真实用户验证的 Zap Bridge 落地页产品，补齐激活漏斗三块短板（激活墙、线索响应闭环、Free 档角色），并用真人冒烟验证全链路。

**总原则：** 本轮不加新区块、不加新模板。只做「验证 + 激活 + 留存」。

---

## 已拍板的产品决策（2026-07-18）

| 决策点 | 结论 |
|---|---|
| 草稿预览链接 | **平台域 + 签名 token**（形如 `zapbridge.tech/preview/<token>`），noindex + 强制水印 + 可过期/撤销。定性为「草稿预览」而非「托管发布」，不违背「不提供共享子域托管」硬约束：临时链接、禁爬、带水印、不可作为投放落地页。 |
| Free 档角色 | **体验漏斗**：保留 1 页 + 水印，**恢复 Meta 基础客户端像素**（`basicPixel: free → true`），让免费用户能跑通小额投放体验核心价值后升级。TikTok/CAPI 等高级追踪维持 Pro/Agency 门控不变。 |
| 域名绑定时机 | 维持「发布必须自有域名」硬约束不动；草稿预览让「见到线上效果」不再被域名前置。 |

## Phase 0（P0，立即，用户主导）：真人生产端到端冒烟

**性质：** 验证任务，无代码改动（发现 bug 则单独开分支修）。
**为什么第一：** 全链路从未被真人在生产走通，任何一环断掉，后续所有开发都不产生用户价值。

**执行清单（在 https://zapbridge.tech 生产环境）：**

- [ ] 1. 新邮箱注册 + Google OAuth 登录各走一遍
- [ ] 2. 选模板建页 → 编辑文案 → 上传图片（验证 Vercel Blob `BLOB_READ_WRITE_TOKEN`）
- [ ] 3. AI 一键生成 / 智能改写各触发一次（验证 `DASHSCOPE_API_KEY` 生产可用 + 次数计数）
- [ ] 4. 绑定 + 验证一个真实自有域名（验证 `VERCEL_API_TOKEN/PROJECT_ID`、A/CNAME 指引对 apex 与子域名都正确）
- [ ] 5. 发布 → 用无痕窗口访问该域名，确认页面、OG 标签、robots/sitemap 正确
- [ ] 6. 在发布页提交一条留资（表单 + WhatsApp 点击）→ 后台「线索」列表可见
- [ ] 7. Meta / TikTok 事件管理后台确认 CAPI 回传事件到达（Pro 档测试账号）
- [ ] 8. **EU 场景合规**：VPN 切欧盟 IP，确认 consent 未同意前像素不注入（CMP 行为）
- [ ] 9. 取消发布 / 删除页面 → 域名与列表状态正确回收

**分工：** 步骤 1–9 需用户真机操作（真实账号、真实域名、广告后台）；AI 侧提前准备核对脚本（`pnpm smoke`）、陪跑排查、每发现一个 bug 按快速通道单独修复。

**验收：** 9 步全绿，或每个失败项都有已建分支的修复记录。

**产出物：** 冒烟结果记录（追加到本文档「冒烟结果」附录节）。

## Phase 1（P0，开发，可与 Phase 0 并行）：草稿分享预览链接 — ✅ 已完成上线（2026-07-19，PR #31 合并 main）

**分支：** `feat_20260718_草稿分享预览链接`（详细计划见 `docs/superpowers/plans/2026-07-18-phase1-草稿分享预览链接.md`）
**目标：** 用户无需绑定域名即可获得一个可分享的线上预览链接，把 aha moment 提前到域名绑定之前；域名绑定从「前置门槛」变为「发布的最后一步」。

**交付结果（PR #31，8 提交，子代理驱动 + 对抗性安全审查 + 代码质量审查）：**
- `migrations/018` 加 `landing_pages.preview_secret`；`lib/preview/token.ts` 签发/校验/撤销/过期（TDD 7 用例）；`getPageForPreview`/`ensurePreviewSecret`/`rotatePreviewSecret`；`POST /api/landing-pages/[id]/preview-link`；`app/preview/[token]/page.tsx`（强制水印+noindex+无追踪）+ 中间件放行 + `next.config` X-Robots-Tag 头；编辑器「分享预览」面板；`e2e/preview-share.spec.ts`。
- 安全审查发现并修复：预览态 `LeadForm` 停用提交（防分享链接灌垃圾线索）。
- 门槛：vitest 202 + tsc 0 + eslint 0 error + build（`/preview/[token]` 注册）+ e2e 连续两次绿。
- 收尾 follow-up（分支 `chore_20260719_预览安全hygiene与路线图`）：① `preview_secret` 经 store `toClient` 从落地页 API 响应剥离，不下发浏览器；② `PUBLIC_PATHS` 改按段边界匹配（消除 `/p` 误伤 `/pricing`/`/preview`，+6 回归测试）。剩余低优先项 `auth.ts` 缺 `AUTH_SECRET` 只 warn 未 fail-closed 记录待办。

**范围与契约方向（细节以该阶段详细计划为准）：**

- 新路由 `app/preview/[token]/page.tsx`：按签名 token 解析出 `landing_page_id`，读草稿数据用现有 `landing-renderer` 渲染。
- Token：HMAC 签名（复用 `AUTH_SECRET` 派生或独立 secret），载荷含 `pageId + 过期时间`；数据库存每页一个可轮换的 `preview_secret`（撤销 = 轮换），需要一个迁移脚本（遵循 `docs/dev-database-migration-workflow.md`）。
- 硬性约束（不可协商）：`X-Robots-Tag: noindex` + meta noindex；**所有档位强制水印**（复用 `landing-renderer/Watermark.tsx`）；默认 7 天过期；不注入任何像素/追踪（预览页不产生投放数据）。
- 编辑器入口：编辑器顶栏加「分享预览」按钮（生成/复制/重置链接），全屏预览页同位复用。
- 中间件：`lib/proxy/auth-proxy.ts` 白名单放行 `/preview/*`；`tenant-proxy` 不得把平台域 `/preview/*` 误判为租户请求（参照 `/robots.txt` 放行先例）。
- 首次发布引导文案同步调整：「先预览分享，绑定域名后正式发布」。

**验收：**
- 未登录访客打开链接可看到与草稿一致的页面，带水印、响应头含 noindex；
- 过期/轮换后旧链接 404；
- 已发布页的自有域名渲染逐字节不受影响（回归测试）；
- E2E：建页 → 生成链接 → 无痕访问 → 断言内容/水印/noindex。

## Phase 2（P1，开发）：线索响应闭环（即时通知 + Webhook）

**分支：** `feat_202607XX_线索通知与Webhook`（开工日定日期）
**目标：** 新线索 5 分钟内触达用户；线索可自动推入用户自己的 CRM。收件箱已有，本阶段补「推」的能力。

**范围与契约方向：**

- **邮件通知（先做）**：新线索落库后经由现有 Resend（`lib/email.ts`）发通知邮件到租户注册邮箱；设置页可开关。发送走异步（落库后触发，失败不阻塞留资接口）。
- **Webhook 推送**：租户在设置页配置一个 URL + 签名密钥；新线索 POST JSON（含 UTM、来源页、字段）；HMAC-SHA256 签名头；失败重试 3 次退避；参考现有 `app/api/webhooks`（LemonSqueezy 入站）的签名习惯做出站对称设计。
- **套餐门控建议**：邮件通知全档位（这是激活体验），Webhook 归 Pro/Agency（集成属高级能力）——此门控在阶段详细计划里再最终确认。
- 不做：WhatsApp/Telegram 通知渠道（依赖第三方 BSP 账号，留下一轮）；Zapier 官方应用（先用通用 Webhook 承接）。

**验收：** 生产提交一条测试线索 → 邮箱收到通知 → 配置的 webhook.site 收到带有效签名的 POST；关闭开关后不再发送。

## 执行规则

- 每个开发阶段：干净工作区 → 从 `main` 按护栏序列建分支 → 写该阶段详细 TDD 计划（`docs/superpowers/plans/`）→ 用户确认 → 实现 → 分层测试 → PR。
- 修 bug（含 Phase 0 冒烟发现的）一律先复现（失败测试/最小脚本）后修复。
- 合并 `dev`/`test`/`main` 由用户手动处理。
- 本文档状态随阶段推进更新（勾选框 + 附录冒烟结果）。
