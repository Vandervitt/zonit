# CI/CD 流程方案（GitHub Actions 质量门禁 + Vercel 原生部署）

分支：`ci_20260714_GitHubActions门禁`　日期：2026-07-14

## 目标

补上 GitHub 侧的 CI 质量门禁,与已有的 Vercel 部署衔接,做到"合并进 main 之前先拦下 lint/类型/单测/构建问题"。

## 已定决策(经用户澄清)

| 维度 | 决策 |
|---|---|
| 部署驱动 | **Vercel 原生 Git 集成**(已在 `vercel.json`),CI 不碰部署 |
| CI 范围 | **快速门禁**:lint + tsc + vitest + next build,无 DB/浏览器 |
| Git 流程 | **PR 门禁**:CI 绿 + Vercel preview 验过才合 main |
| CI 保留 build | 是(GitHub 原生 required check 更硬,用假 env 跑通) |
| E2E / Rolling Releases / GHA 驱动部署 | 均不做(YAGNI) |

## 背景事实(探索所得)

- 无 `.github/workflows`,CI 空白。
- `vercel.json` 已配 `git.deploymentEnabled`(main + preview)与 cron;`vercel-build` = `pnpm migrate:up && next build`(生产部署时自动迁移库)。
- 本地未 `vercel link`(无 `.vercel`)。
- 脚本齐:`lint` / `test`(vitest 89)/ `test:e2e` / `migrate` / `check:env`。
- 项目**无 generateStaticParams、无静态导出、无构建期 DB 访问** → `next build` 不需真实 DB,仅需导入期被读的 env 占位。
- 版本:Node 22.20.0、pnpm 11.3.0、lockfile v9、Next 16.2.3。`packageManager`/Node 版本此前未声明。

## 架构(职责分离)

```
开发者 → 功能分支 → push
                     ├─ GitHub Actions CI (质量门禁: lint/tsc/test/build)
                     └─ Vercel Preview 部署 (真实构建 + 预览 URL)
   PR → 两个 check 都绿 → 合并 main
                          └─ Vercel 生产部署 (vercel-build: migrate + build)
```

- **CD** 全交给 Vercel 原生集成,不写部署代码。
- **CI** 一个 workflow,只做门禁。

## `.github/workflows/ci.yml` 设计

- **name**: CI
- **on**: `pull_request`(target main)+ `push`(branches: main)
- **concurrency**: group 按 `${{ github.workflow }}-${{ github.ref }}`,`cancel-in-progress: true`(同分支新 push 取消旧运行)
- **权限**: `contents: read`(最小)
- **单 job `verify`**(ubuntu-latest,顺序步骤):
  1. `actions/checkout@v4`
  2. `pnpm/action-setup@v4`(读 package.json 的 packageManager,或显式 version: 11.3.0)
  3. `actions/setup-node@v4`(node-version-file: `.nvmrc`,cache: `pnpm`)
  4. `pnpm install --frozen-lockfile`
  5. `pnpm lint`
  6. `pnpm exec tsc --noEmit`
  7. `pnpm test`(vitest run)
  8. `pnpm build`,env 注入**假构建值**(不连库):
     - `AUTH_SECRET`: 32+ 位假串
     - `NEXTAUTH_URL` / `NEXT_PUBLIC_APP_URL`: `http://localhost:3001`
     - 其余若构建期报缺,再按需补假值(原则:只补构建期真正读取的,不塞运行时密钥)

> 步骤顺序遵循项目分层验证:lint → 类型 → 单测 → 构建。任一步失败即红。

## PR 门禁生效(GitHub 仓库设置,非代码)

CI workflow 合入后,在 GitHub 仓库执行(二选一):

- **UI**:Settings → Branches → Add branch ruleset / protection rule,target `main`:
  - Require a pull request before merging
  - Require status checks to pass → 勾选 `CI / verify`
  - (可选)Require branches to be up to date before merging
- **CLI**(需本机装 `gh` 且有权限)：
  ```
  gh api -X PUT repos/Vandervitt/zapbridge/branches/main/protection \
    -f 'required_status_checks[strict]=true' \
    -f 'required_status_checks[checks][][context]=CI / verify' \
    -F 'enforce_admins=true' \
    -F 'required_pull_request_reviews[required_approving_review_count]=0' \
    -F 'restrictions='
  ```
  > 注:`gh` 当前本机未安装;首次需 CI 至少跑过一次,check 名字才会出现在可选列表。

## Vercel 侧(用户确认,不在本次代码范围,涉及密钥)

- 确认 Vercel 项目已连该 GitHub 仓库(Git Integration)。
- 生产环境变量按 `scripts/check-env.mjs` 必填清单在 Vercel 配好:
  `DATABASE_URL` / `DATABASE_URL_UNPOOLED` / `AUTH_SECRET` / `NEXTAUTH_URL` /
  `AUTH_GOOGLE_ID` / `AUTH_GOOGLE_SECRET` / `VERCEL_API_TOKEN` / `VERCEL_PROJECT_ID` /
  `VERCEL_TEAM_ID` / `BLOB_READ_WRITE_TOKEN` / `NEXT_PUBLIC_APP_URL` / `CRON_SECRET`。
- 生产部署用 `vercel-build`,会先 `migrate:up`,故 `DATABASE_URL_UNPOOLED` 必配。

## 验证方式

- 本地:`pnpm lint` / `pnpm exec tsc --noEmit` / `pnpm test` 均已知通过(89/89)。
- `pnpm build`:本地受沙箱字体下载网络墙阻断;**CI(GitHub runner 有网)可正常验证** —— 以 workflow 首次实跑为准。
- workflow 语法:用 `actionlint`(若可得)或推送后看 Actions 首跑结果。
- 门禁有效性:开一个含故意 lint 错误的测试 PR,确认 CI 变红并阻止合并(可选演示)。

## 风险与取舍

- **`next build` 假 env**:若构建期意外读取某运行时密钥导致失败,按"只补构建期真正读取的假值"原则最小补齐;不塞真实密钥进 CI。
- **build 与 Vercel preview 重叠**:有意保留,换取 GitHub 原生 required check;成本是每次多跑一次构建(~1-2 分钟)。
- **分支保护是仓库设置**:无法由本仓库代码提交生效,需人工在 GitHub 开启(已给 UI/CLI 两法)。

## Vercel preview 部署(CD 侧,独立问题)
